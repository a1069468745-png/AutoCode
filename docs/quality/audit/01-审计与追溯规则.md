# 审计与追溯规则

## 1. 审计目标

确保每项重要工作都能回答：

- 为什么做
- 依据什么做
- 改了什么
- 如何验证
- 当前状态是什么

## 2. 最小追溯链

每项重要工作至少要能追溯到：

- 任务或计划
- 相关设计或需求
- 实现变更
- 验证结果
- 进度记录

## 3. 必须回写的场景

- 目录结构变化
- 治理规则变化
- 架构决策变化
- 实施进度推进
- 发布门禁结果
- 任务边界或进入门禁定义变化
- 阶段自测边界或测试用例补充

## 4. 推荐记录位置

- 进度类：`docs/context/progress/`
- 决策类：`docs/context/decisions/`
- 任务状态类：`docs/context/current/02-当前任务状态.md`

## 5. 审计优先级

- 高优先级：架构、门禁、计划、发布
- 中优先级：模块实现、测试策略、索引规则
- 低优先级：普通描述优化、格式清理

## 6. 文档更新审计要求

- 必须说明为什么更新这些文档
- 必须说明为什么没有更新其他关联文档，若适用
- 不得出现无边界、无依据的大范围文档修改
- 审计结果、发布门禁结果等结果性文件不应长期与本规则文件同层混放，应优先归档或通过索引进入

## 7. 结果文件命名与存放约束

- 规则类文件保留稳定序号命名
- 审计结果、发布门禁结果等结果性文件应使用带日期命名
- 结果性文件优先进入 `archive/` 或由索引文件统一指向，不直接暴露在主导航热路径

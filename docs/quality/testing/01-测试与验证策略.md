# 测试与验证策略

## 1. 总体原则

- 没有验证策略的任务，不进入开发执行
- 没有验证结果的任务，不宣告完成
- 高风险改动优先自动化验证
- 进入阶段实施前，必须存在阶段自测边界、测试用例或明确验收边界
- 如果阶段自测边界缺失，先补边界和用例，再进入阶段实施

## 2. 验证类型

- 构建验证
- 单元测试
- 集成测试
- 样例数据人工核验
- 文档一致性核验

## 3. 任务级要求

- 小改动：至少构建通过或目标测试通过
- 模块改动：至少单元测试或集成验证通过
- 跨模块改动：至少联调验证通过
- 架构或治理改动：至少文档一致性与引用链校验通过

## 4. 阶段级要求

- 进入新阶段前，必须明确阶段目标对应的验证边界
- 阶段主链路必须至少有一组可执行或可核验用例
- 阶段验收不得临时发挥，必须基于事先定义的边界执行

## 5. 结果记录

验证结果必须写入：

- 当前会话回复
- 必要时写入 `docs/context/progress/`

## 6. 特殊情况

如果当前环境无法执行测试，必须明确记录：

- 未执行原因
- 潜在风险
- 后续补测要求

# 索引任务运行监控与告警规范 (OPS-03)

## 1. 目标
定义一期 MVP 索引任务的监控看板字段、告警规则与故障定位流程。

## 2. 监控字段

### 2.1 项目级任务状态（实时）
通过 `GET /api/projects/{id}` 获取：

| 字段 | 来源 | 说明 |
|------|------|------|
| `status` | `app.projects.status` | CREATED / INDEXING / READY / FAILED |
| `indexError` | `app.projects.index_error` | 失败时包含错误原因（截断至 2000 字符） |

### 2.2 索引任务详情（通过各服务查询）
- CodeGraph: `POST /api/runner/codegraph/index` 返回 `symbolCount` + `edgeCount`
- History: `POST /api/runner/history/index` 返回 `commitCount` + `linkCount`
- Knowledge: `POST /api/runner/knowledge/index` 返回 `documentCount` + `requirementCount` + `linkCount`

## 3. 告警规则

| 规则 | 条件 | 级别 |
|------|------|------|
| 索引失败 | `status = FAILED` | P2（需人工介入） |
| 索引超时 | `status = INDEXING` 持续 > 30 分钟 | P3（可能阻塞） |
| 符号数为零 | `symbolCount = 0` 且项目含源码文件 | P3（可能配置错误） |
| 提交数为零 | `commitCount = 0` 且项目有 git 历史 | P3（可能工作区路径错误） |

## 4. 故障定位流程

1. 检查 `app.projects.index_error` 获取错误原因
2. 检查各服务日志：`autocode-index-*` 线程前缀
3. 验证工作区路径存在且有正确权限
4. 验证 PostgreSQL 连接正常
5. 必要时手动调用对应 `/api/runner/*/index` 排查

## 5. 长耗时识别
- 索引线程使用 `autocode-index-` 前缀，可通过线程名过滤
- 日志中记录开始/完成时间戳，计算耗时
- 建议在项目详情 API 中增加 `index_started_at` 字段用于计算耗时

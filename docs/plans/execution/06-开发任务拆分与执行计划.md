# 历史代码智能分析与智能开发平台开发任务拆分与执行计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 基于统一终态技术栈，分阶段交付一套可内网部署、多人共用、支持历史分析、调用关系分析、文档关联检索和后续智能开发辅助的平台。

**Architecture:** 平台采用前后端分离和在线服务/后台任务分离架构。所有核心组件从第一阶段按终态方案一次定型，后续阶段只新增能力启用，不替换核心组件或重建底座。

**Tech Stack:** `Java 21`、`Spring Boot 3.3+`、`Vue 3 + TypeScript + Element Plus`、`PostgreSQL 15+`、`Redis 7`、`Qdrant`、`CodeGraph`、`JGit`、`Obsidian + Git Markdown`、`Docker Compose`、`Nginx`

---

## 1. 执行原则

- 所有任务以终态模块为边界拆分，不按临时方案拆分。
- 所有模块先完成接口、数据模型、部署骨架，再逐步启用业务能力。
- 每个任务必须具备清晰边界、输入输出、门禁和后续依赖。
- 门禁未通过的任务不得进入后续任务。
- `MVP` 仅表示第一阶段必须完成，不表示技术方案降级。
- 涉及跨模块联调的任务，必须在单模块门禁全部通过后启动。

## 3. 门禁定义

### 3.1 设计门禁

- 接口定义完成并评审通过。
- 表结构和索引设计完成并评审通过。
- 配置项、错误码、日志字段和安全约束已明确。

### 3.2 开发门禁

- 核心功能代码完成。
- 单元测试通过。
- 异常路径具备可验证行为。
- 日志、配置、健康检查已接入。

### 3.3 联调门禁

- 上下游接口联通。
- 样例数据跑通核心链路。
- 核心查询链路结果可人工校验。

### 3.4 发布门禁

- Docker 镜像可构建。
- Docker Compose 可启动。
- 健康检查、日志和基础监控可用。
- 验收清单通过。

## 9. 任务边界控制规则

- 一个任务只交付一个可验证目标，不混合多个模块主责任。
- 任何任务如果需要新增表结构或接口，必须先回到设计门禁。
- 任何任务如果依赖上游服务真实输出，必须等待其联调门禁通过。
- 所有“智能”相关任务必须建立在结构化查询链路稳定之后，不得提前穿透到底层模型调用。

## 10. MVP 完成定义

满足以下条件时，一期 `MVP` 可视为完成：

- 平台可在内网通过 Docker Compose 启动。
- 用户可登录并按项目访问。
- 管理员可接入一个真实项目。
- 系统可完成代码图谱索引、Git 历史索引、文档索引。
- 用户可查询调用关系、历史变更和文档关联。
- 前端可展示项目切换、查询结果、调用链图和历史详情。
- 审计、健康检查、日志和基础部署文档完整。
- `llm-gateway` 和 `qdrant` 已完成终态接口预留和部署预留。

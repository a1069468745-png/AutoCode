# 文档治理瘦身与上下文加载优化设计

**日期：** `2026-05-29`

## 1. 背景

当前仓库已经形成 `indexes / governance / specs / plans / quality / context` 六层文档结构，并明确要求新会话采用最小必要上下文加载方式启动。这一方向是正确的，但在实际使用中，多个入口文件对相同规则进行了重复描述，导致：

- 新会话为保持合规而重复读取相同约束
- 当前状态、导航索引、治理规则之间存在信息重叠
- Agent 需要花费额外 token 判断“哪份文档才是规范源”
- 动态状态与静态规则混放，增加上下文噪音

本设计的目标不是弱化治理，而是在不降低约束强度的前提下，压缩重复入口、统一规范源、降低会话 token 消耗。

## 2. 设计目标

### 2.1 必须达成

- 保留现有强约束治理能力
- 保留最小启动集和按任务类型追加加载机制
- 保留门禁、验证、回写、审计的完整链路
- 降低新会话和连续会话恢复时的上下文冗余
- 明确每类文档的唯一职责，减少交叉重复

### 2.2 明确不做

- 不重构现有六层目录体系
- 不取消 `AGENT.md + CLAUDE.md` 双入口模式
- 不取消 `docs/context/current/` 作为当前事实来源
- 不用聊天记忆替代仓库文档

## 3. 必须保留的硬约束

以下约束属于仓库治理骨架，不能因节省 token 而删除，只能集中定义、减少重复转述：

1. 权威来源顺序与冲突处理规则
2. 新会话最小启动集
3. 先计划、后开发
4. 任务边界与进入门禁检查
5. 阶段自测边界或验收边界要求
6. 完成声明必须具备验证证据
7. 重要工作必须回写进度、状态、必要决策
8. 当前状态以 `docs/context/current/` 为准
9. 治理、计划、设计、测试、审计等文档默认使用中文

## 4. 核心问题归纳

### 4.1 重复规则分散在多个入口

“新会话最小启动集”“先计划后开发”“回写与审计”等规则同时出现在以下文件中：

- `AGENT.md`
- `CLAUDE.md`
- `docs/indexes/00-总索引.md`
- `docs/governance/03-会话与上下文加载规则.md`
- `README.md`

这些规则本身需要存在，但不需要在多个入口重复完整正文。

### 4.2 当前索引与当前状态有重复事实

以下文件之间存在明显重叠：

- `docs/indexes/02-当前阶段索引.md`
- `docs/indexes/03-当前任务索引.md`
- `docs/context/current/01-当前阶段与目标.md`
- `docs/context/current/02-当前任务状态.md`

索引本应只承担导航职责，不应重复承载完整事实。

### 4.3 静态规则与动态状态混杂

部分入口文档既写长期适用的治理规则，又写当前阶段进展和实施顺序，导致：

- 静态规则文件频繁变化
- 动态状态信息在多个位置维护
- Agent 难以判断“这份文档是规则还是现状”

### 4.4 长文档缺少读取决策摘要

规范类文档虽然不算极长，但大多缺少统一的顶部摘要。Agent 往往需要先读正文，才能确认是否应继续深入。

## 5. 推荐方案

采用“结构收敛型”方案：保留现有六层目录，但明确每层只承担一种职责，并将重复规则改为“单点定义 + 交叉引用”。

## 6. 目标信息分层

优化后，仓库中的信息应严格按以下原则分层：

- 规则定义只在 `docs/governance/` 出现
- 会话入口只在 `docs/indexes/` 出现
- 当前事实只在 `docs/context/current/` 出现
- 历史演进只在 `docs/context/progress/` 出现
- 审计留痕只在 `docs/quality/` 中定义
- 仓库对外介绍只在 `README.md` 出现

## 7. 目标文档职责

### 7.1 `AGENT.md`

只保留：

- 最高优先级原则
- 权威来源顺序
- 硬边界与禁止事项
- 文档语言规范
- 对最小加载规则的引用

不再保留：

- 详细按任务类型加载正文
- 细粒度门禁清单
- 当前阶段或当前任务动态信息

### 7.2 `CLAUDE.md`

只保留：

- Agent 执行协议
- 任务类型分流
- 实施前检查框架
- 验证与回写协议
- 对正式门禁规范的引用

不再保留：

- 与 `AGENT.md` 完全同义的原则列表
- 与 `governance/03` 完全重复的加载正文

### 7.3 `docs/indexes/00-总索引.md`

只保留：

- 新会话最小启动集
- 文档目录的一句话职责说明
- 按任务类型跳转建议

不再保留：

- 治理规则解释性正文
- 当前阶段与当前任务的展开描述

### 7.4 `docs/indexes/02-当前阶段索引.md`

压缩为轻量快照，只保留：

- 当前阶段
- 阶段目标一句话
- 下一阶段
- 关联状态文档
- 关联计划入口

### 7.5 `docs/indexes/03-当前任务索引.md`

压缩为任务快照卡，只保留：

- 当前主任务
- 当前优先任务编号或名称
- 当前状态一句话
- 门禁摘要一句话
- 继续本任务建议读取的文档列表

### 7.6 `docs/context/current/01-当前阶段与目标.md`

作为当前阶段事实唯一来源，保留完整阶段目标、完成标准、阶段说明。

### 7.7 `docs/context/current/02-当前任务状态.md`

作为当前任务事实唯一来源，保留：

- 当前状态
- 已完成
- 待完成
- 风险
- 下一步建议
- 最近验证结论摘要

### 7.8 `docs/governance/01-开发治理规则.md`

只定义治理原则，不再重复加载入口和当前状态内容。

### 7.9 `docs/governance/02-门禁与审计规则.md`

聚焦“能不能做”的规则，建议改为表格化清单，作为唯一门禁规范源，覆盖：

- 设计门禁
- 开发门禁
- 阶段进入门禁
- 测试门禁
- 发布门禁
- 门禁失败处理

### 7.10 `docs/governance/03-会话与上下文加载规则.md`

作为唯一加载规范源，定义：

- 默认最小加载集
- 按任务类型追加加载
- 长文档按章节读取原则
- 连续会话恢复策略

### 7.11 `docs/quality/audit/01-审计与追溯规则.md`

聚焦“做完如何留痕”，只保留：

- 最小追溯链
- 必须回写场景
- 推荐记录位置
- 审计优先级
- 文档更新审计要求

不再重复门禁判断正文。

### 7.12 `README.md`

回到对外入口角色，只保留：

- 项目简介
- 仓库结构
- 当前文档入口
- 开发前建议读取路径

不再承载详细动态进度与任务顺序。

## 8. 标准摘要头模板

建议为治理、质量、计划、设计类长文档统一增加摘要头，帮助 Agent 在最少读取量下判断是否继续深入：

```md
## 文档摘要
- 适用任务：
- 何时必须读取：
- 可跳过内容：
- 关联权威文档：
- 更新触发条件：
```

该摘要头不替代正文，只用于降低读取决策成本。

## 9. 推荐落地顺序

### 第一阶段：入口瘦身

优先调整以下文件：

- `README.md`
- `docs/indexes/00-总索引.md`
- `docs/indexes/02-当前阶段索引.md`
- `docs/indexes/03-当前任务索引.md`

目标是让会话入口只回答“现在该读什么”，不再重复治理正文。

### 第二阶段：协议收敛

调整以下文件：

- `AGENT.md`
- `CLAUDE.md`

目标是保留双入口，但减少双份完整规则说明。

### 第三阶段：规则去重

调整以下文件：

- `docs/governance/02-门禁与审计规则.md`
- `docs/quality/audit/01-审计与追溯规则.md`

目标是将“门禁判断”和“审计留痕”明确拆开。

### 第四阶段：长文档摘要化

为治理、质量、计划、设计文档批量补齐摘要头，降低后续读取成本。

## 10. 预期收益

优化完成后，预期获得以下收益：

- 新会话默认加载成本下降
- Agent 更容易判断单一规范源
- 当前事实与导航索引边界更清晰
- 审计与回写要求保持不变，但更容易执行
- 动态状态的维护成本下降
- 后续文档扩张时更不容易重新回到重复堆叠模式

## 11. 风险与约束

### 11.1 风险

- 如果精简过度，可能误伤治理强度
- 如果只删内容不重建引用关系，可能导致入口断裂
- 如果索引压缩后未补充跳转路径，可能降低可发现性

### 11.2 控制措施

- 只删除重复表述，不删除治理骨架
- 每处精简都要明确新的规范源位置
- 动态状态只迁移，不丢失
- 所有改动以“单点定义、其他引用”为原则

## 12. 结论

本次优化不应追求“文档越少越好”，而应追求“每种信息只出现一次，且能被最低成本找到”。在此原则下，最适合当前仓库的方案是：

- 保留六层结构
- 保留双入口模式
- 保留最小启动集与强门禁机制
- 压缩索引与入口重复
- 将规则、导航、当前事实、历史演进彻底分层

该方案能够在不削弱治理、计划、审计能力的前提下，显著改善文档加载成本与上下文效率。

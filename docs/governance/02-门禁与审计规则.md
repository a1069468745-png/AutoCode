# 门禁与审计规则

## 1. 设计门禁

必须满足：

- 任务边界明确
- 上游权威文档已读取
- 接口、表结构、配置项、日志字段有依据
- 已明确当前任务的进入门禁条件

## 2. 开发门禁

必须满足：

- 当前任务有明确计划
- 当前任务有明确边界
- 上游门禁已完成
- 当前任务进入门禁已检查并通过
- 本次研发内验证方式已知
- 本次最小回写目标已知

说明：

- 开发门禁只约束当前研发阶段的实施与自测
- 评审、审计、发布要求在进入对应阶段时再生效

## 3. 阶段进入门禁

必须满足：

- 当前阶段目标明确
- 当前阶段存在自测边界、测试用例或明确验收边界
- 阶段内关键链路具备验证路径

如果当前阶段缺少自测边界，必须先补充，再进入阶段实施。

## 4. 研发内自测门禁

必须满足：

- 至少有一种验证方法
- 关键路径有测试或人工验证依据
- 风险点有明确检查项

说明：

- 本节用于约束研发阶段的最小自测闭环
- 不要求在开发阶段预先补齐评审结论、完整审计材料或发布材料

## 5. 评审门禁

进入评审阶段时必须满足：

- 已有上游计划依据
- 已有对应实现或文档更新
- 已有验证结论或明确风险说明
- 已完成研发阶段要求的最小回写

## 6. 发布门禁

必须满足：

- 构建可通过
- 关键服务可启动
- 核心能力可演示
- 进度、审计、验收记录完整

## 7. 审计留痕要求

关键工作必须可追溯到：

- 任务编号或任务名称
- 相关计划文档
- 相关实现变更
- 相关验证结论
- 必要的状态或进度回写记录

说明：

- 研发阶段保留最小留痕链路即可
- 完整审计材料在审计或发布阶段按需补齐

## 8. 文档更新边界规则

- 文档更新必须基于任务边界和实际影响范围
- 需要更新的文档不得遗漏
- 无关文档不得随意修改
- 如果发现应更新范围超出当前任务边界，必须先回到计划或决策层对齐

## 9. 失败处理

如果门禁未通过：

- 停止进入下游任务
- 记录阻塞原因
- 更新当前任务状态
- 必要时补充决策或计划文档

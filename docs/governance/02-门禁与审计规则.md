# 门禁与审计规则

## 1. 设计门禁

必须满足：

- 任务边界明确
- 上游权威文档已读取
- 接口、表结构、配置项、日志字段有依据
- 已明确当前任务的进入门禁条件

## 2. 开发门禁

必须满足：

- 当前任务有明确计划
- 当前任务有明确边界
- 上游门禁已完成
- 当前任务进入门禁已检查并通过
- 验证方式已知
- 本次改动要更新的文档已知

## 3. 阶段进入门禁

必须满足：

- 当前阶段目标明确
- 当前阶段存在自测边界、测试用例或明确验收边界
- 阶段内关键链路具备验证路径

如果当前阶段缺少自测边界，必须先补充，再进入阶段实施。

## 4. 测试门禁

必须满足：

- 至少有一种验证方法
- 关键路径有测试或人工验证依据
- 风险点有明确检查项

## 5. 发布门禁

必须满足：

- 构建可通过
- 关键服务可启动
- 核心能力可演示
- 进度、审计、验收记录完整

## 6. 审计留痕要求

关键工作必须可追溯到：

- 任务编号或任务名称
- 相关计划文档
- 相关实现变更
- 相关验证结果
- 进度回写记录

## 7. 文档更新边界规则

- 文档更新必须基于任务边界和实际影响范围
- 需要更新的文档不得遗漏
- 无关文档不得随意修改
- 如果发现应更新范围超出当前任务边界，必须先回到计划或决策层对齐

## 8. 失败处理

如果门禁未通过：

- 停止进入下游任务
- 记录阻塞原因
- 更新当前任务状态
- 必要时补充决策或计划文档

# 开发治理规则

## 1. 基本原则

- 先计划，后开发
- 先检查任务边界与进入门禁，后启动任务
- 先检查阶段自测边界，后进入阶段实施
- 先边界，后实现
- 先验证，后完成
- 先回写，后切换任务
- 门禁按阶段激活，研发阶段不前置评审、审计、发布要求

## 2. 允许的工作方式

- 基于执行计划中的任务编号开展工作
- 在任务边界内做最小必要实现
- 在设计、实现、研发内自测、评审、审计之间保持可追溯链路
- 按任务实际影响范围更新必要文档与进度文档

## 3. 禁止的工作方式

- 无任务边界直接编码
- 未检查进入门禁直接启动任务
- 阶段自测边界缺失时直接推进阶段开发
- 无验证直接宣告完成
- 无文档回写直接结束长任务
- 随意更新无关文档或超出边界回写
- 用临时聊天结论替代文档事实
- 未过门禁提前进入下游任务
- 在研发阶段强制前置评审、审计或发布材料，造成无必要阻塞

## 4. 文档遵循规则

- 范围问题以 PRD 为准
- 技术方案以 HLD/LLD 为准
- 顺序和拆分以执行计划为准
- 当前状态以 `docs/context/current/` 为准

## 5. 变更规则

- 改产品范围，更新 `docs/specs/product/`
- 改架构，更新 `docs/specs/architecture/` 与 `docs/context/decisions/`
- 改执行路径，更新 `docs/plans/execution/`
- 改当前进度，至少更新 `docs/context/current/`；如属于阶段性推进，再更新 `docs/context/progress/`

## 6. 文档语言规范

- 后续新增文档默认使用中文
- 需求、设计、治理、计划、测试、评审、审计、进度、决策类文档必须默认使用中文
- 技术名词、接口名、协议名、字段名、代码标识可在必要时保留英文
- 不得无理由新增英文长篇治理或项目文档

## 7. 文档准入与命名规则

新增文档前，必须先判断它属于以下哪一类：

- 当前事实：进入 `docs/context/current/`
- 历史进度：进入 `docs/context/progress/`
- 持久决策：进入 `docs/context/decisions/`
- 长期规则或策略：进入 `docs/governance/` 或 `docs/quality/`
- 阶段性清单、验收结果、评审结果、审计结果：优先进入对应目录的 `archive/` 或通过索引进入

不得新增以下类型的噪音文档：

- 与 `current/` 重复维护当前状态的进度文档
- 与正式规范重复的过程产物说明
- 已过期但仍暴露在主导航中的阶段性结果文件
- 将短期推进状态写成持久决策的文档

推荐命名模式：

- 进度：`YYYY-MM-DD-<任务编号或主题>-进度.md`
- 决策：`YYYY-MM-DD-<主题>-决策.md`
- 阶段性结果：`archive/YYYY-MM-DD-<主题>-结果.md` 或保留原主题名后放入 `archive/`
- 长期策略或规则：使用稳定序号命名，如 `01-*.md`

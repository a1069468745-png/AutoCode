# 开发治理规则

## 1. 基本原则

- 先计划，后开发
- 先检查任务边界与进入门禁，后启动任务
- 先检查阶段自测边界，后进入阶段实施
- 先边界，后实现
- 先验证，后完成
- 先回写，后切换任务

## 2. 允许的工作方式

- 基于执行计划中的任务编号开展工作
- 在任务边界内做最小必要实现
- 在设计、实现、测试、审计之间保持可追溯链路
- 按任务实际影响范围更新必要文档与进度文档

## 3. 禁止的工作方式

- 无任务边界直接编码
- 未检查进入门禁直接启动任务
- 阶段自测边界缺失时直接推进阶段开发
- 无验证直接宣告完成
- 无文档回写直接结束长任务
- 随意更新无关文档或超出边界回写
- 用临时聊天结论替代文档事实
- 未过门禁提前进入下游任务

## 4. 文档遵循规则

- 范围问题以 PRD 为准
- 技术方案以 HLD/LLD 为准
- 顺序和拆分以执行计划为准
- 当前状态以 `docs/context/current/` 为准

## 5. 变更规则

- 改产品范围，更新 `docs/specs/product/`
- 改架构，更新 `docs/specs/architecture/` 与 `docs/context/decisions/`
- 改执行路径，更新 `docs/plans/execution/`
- 改当前进度，更新 `docs/context/progress/` 与 `docs/context/current/`

## 6. 文档语言规范

- 后续新增文档默认使用中文
- 需求、设计、治理、计划、测试、评审、审计、进度、决策类文档必须默认使用中文
- 技术名词、接口名、协议名、字段名、代码标识可在必要时保留英文
- 不得无理由新增英文长篇治理或项目文档

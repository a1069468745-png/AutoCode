# 历史代码智能分析与智能开发平台需求文档（PRD）

## 1. 文档信息

- 项目名称：历史代码智能分析与智能开发平台
- 文档类型：产品需求文档（PRD）
- 版本：V1.0
- 适用范围：内网服务器部署、多人共用、离线可运行的软件研发团队

## 2. 项目背景

企业在长期研发过程中，会积累大量历史代码、历史需求、设计文档、缺陷记录和发布记录。随着系统规模扩大、人员流动增加、技术栈演进加快，团队普遍面临以下问题：

- 历史代码理解成本高，新成员难以快速定位关键逻辑。
- 调用关系不透明，影响范围分析依赖少数核心人员经验。
- 历史需求、设计文档与代码实现脱节，无法形成有效知识闭环。
- 新需求开发前缺乏系统化的上下文准备，AI 编码容易脱离真实架构。
- 多项目并行时，团队难以实现统一的知识复用与智能辅助开发。

为解决上述问题，需建设一套面向内网、多项目、可离线部署的平台，统一承载历史代码理解、调用关系分析、历史知识关联和后续智能开发辅助能力。

## 3. 建设目标

### 3.1 总体目标

建设一套可在企业内网部署的智能研发辅助平台，实现：

- 对历史代码进行结构化理解与检索。
- 对函数、类、接口、模块提供调用关系与影响范围分析。
- 对 Git 历史变更进行结构化分析与追溯。
- 对需求、设计、缺陷、发布等文档形成知识库并与代码关联。
- 在此基础上支持新需求的智能分析、改造建议与辅助开发。

### 3.2 分阶段目标

#### 第一阶段

- 实现历史代码索引、符号检索、调用关系分析。
- 实现 Git 历史分析与关键改动追溯。
- 实现按项目切换的统一查询能力。

#### 第二阶段

- 建立需求、设计文档与代码的关联关系。
- 支持从需求定位代码，从代码回溯设计与历史变更。

#### 第三阶段

- 支持新需求输入后的上下文自动组织。
- 支持影响范围分析、改造方案建议、测试建议生成。
- 支持接入第三方大模型 API 和本地模型。

## 4. 用户角色

- 架构师：关注系统结构、模块边界、依赖关系、改造影响。
- 开发工程师：关注功能入口、调用链、历史改动、新需求改造建议。
- 测试工程师：关注变更影响范围、受影响接口、受影响测试集。
- 技术经理：关注知识沉淀、团队协作效率、项目接入成本、风险控制。
- 运维/平台工程师：关注部署方式、权限隔离、资源使用、模型接入策略。

## 5. 核心业务场景

### 5.1 历史代码理解

用户输入自然语言问题，例如“退款流程入口在哪”“登录鉴权链路是什么”，系统返回：

- 相关模块
- 关键入口函数
- 上下游调用关系
- 相关设计文档
- 历史变更摘要

### 5.2 调用关系分析

用户选择某个函数、接口或类，系统返回：

- 调用者列表
- 被调用者列表
- 跨模块依赖链
- 影响半径

### 5.3 历史分析

用户查看某段代码、某模块或某功能的历史信息，系统返回：

- 首次引入时间
- 关键修改记录
- 相关提交与责任人
- 关联需求与缺陷
- 高频改动位置

### 5.4 新需求智能分析

用户输入新需求，系统自动输出：

- 疑似涉及模块
- 相关历史需求
- 关键代码入口
- 影响范围
- 推荐改造方案
- 推荐测试点

## 6. 功能需求

### 6.1 项目管理

- 支持新增项目、编辑项目、禁用项目。
- 支持配置代码仓库地址、默认分支、技术栈、文档仓路径。
- 支持为项目配置默认模型、备用模型和模型访问策略。
- 支持项目级数据隔离与权限控制。

### 6.2 代码图谱与调用分析

- 支持按项目建立代码索引。
- 支持符号级检索，包括函数、类、接口、类型、路由、组件。
- 支持展示 callers、callees、imports、inheritance、references。
- 支持查看影响范围和调用链路。
- 支持增量更新索引。

### 6.3 Git 历史分析

- 支持解析 commit、diff、blame、tag、release。
- 支持将提交与文件、符号、模块关联。
- 支持识别需求编号、缺陷编号、版本号。
- 支持按时间、模块、作者、需求查看变更历史。

### 6.4 文档知识库

- 支持接入 Markdown 格式知识库。
- 支持按项目管理需求文档、概要设计、详细设计、ADR、复盘等文档。
- 支持文档元数据解析与索引。
- 支持文档与代码符号、路径、提交进行关联。

### 6.5 统一检索与问答

- 支持自然语言检索代码、历史、文档信息。
- 支持结构化查询与自然语言问答结合。
- 支持项目维度上下文切换。
- 支持按查询类型自动路由到代码图谱、历史索引或文档索引。

### 6.6 智能开发辅助

- 支持新需求解析。
- 支持相似需求推荐。
- 支持自动构建相关上下文。
- 支持生成改造方案、影响点清单、测试建议。
- 支持与外部编码代理协同工作。

### 6.7 模型接入

- 支持配置第三方模型 API。
- 支持统一 API 网关。
- 支持本地离线模型与外部模型并存。
- 支持按项目切换模型。
- 支持模型熔断、超时、回退与审计。

## 7. 非功能需求

### 7.1 部署要求

- 支持内网服务器部署。
- 支持无外网情况下运行。
- 支持 Docker Compose 起步，后续可迁移 Kubernetes。

### 7.2 性能要求

- 已接入项目切换响应时间应控制在秒级。
- 历史索引与代码图谱查询应满足日常研发交互需求。
- 支持中小规模团队并发使用。

### 7.3 安全要求

- 代码与文档数据不得默认出网。
- 模型调用策略可配置。
- 支持访问鉴权、项目权限控制、审计日志。

### 7.4 可维护性要求

- 各核心模块解耦。
- 支持新项目快速接入。
- 支持后续增加新的代码分析引擎、模型供应商、检索策略。

## 8. 成功标准

- 新成员理解核心系统功能的时间明显缩短。
- 常见“入口在哪、谁调用谁、改动影响什么”问题可由平台直接回答。
- 新需求分析前置成本降低。
- 历史文档与历史代码的复用率提升。
- AI 编码建议与实际代码架构的一致性提升。

## 9. 范围边界

### 9.1 本期范围

- 历史代码索引
- 调用关系分析
- Git 历史分析
- 文档知识库接入
- 项目切换与统一查询
- 智能开发辅助前置能力

### 9.2 非本期范围

- 全自动无人工确认的生产级改代码闭环
- 自动发布与上线审批
- 覆盖所有业务流程的流程编排平台能力

- 需求文档
- 概要设计
- 详细设计
- 风险分析报告
- 痛点分析报告
